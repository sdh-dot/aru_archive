# 루리웹 DOM 동작 관찰 기록 (Phase 1)

본 문서는 Phase 1 사전 조사에서 확인된 루리웹 글쓰기 페이지의 DOM 동작을 기록합니다. 이후 루리웹 측 변경이 발생하면 회귀 비교용 baseline으로 사용합니다.

> 관찰 대상: 루리웹 유게 잡담 게시판 글쓰기 페이지 본문 에디터.

## 관찰된 사실 (8항)

### 1. 이미지는 `p > img` 구조로 삽입된다

업로드된 이미지는 다음과 같은 형태로 본문 에디터에 들어갑니다.

```html
<p>
  <img src="https://image.ruliweb.com/.../filename.webp" alt="원본파일명.jpg">
</p>
```

이미지는 항상 `<p>` 안에 단독으로 들어가며, 캡션 삽입 위치는 이 `<p>` 바로 다음 형제입니다.

### 2. `img.src`는 루리웹 CDN WebP URL로 변환된다

업로드 시 루리웹은 원본 파일을 자체 CDN의 WebP 포맷으로 변환하고 그 URL을 `src`에 넣습니다. 따라서 `src`는 원본 파일과 직접 매칭할 수 없습니다.

```text
변환 전 (가상): src 값이 원본 파일명을 포함
변환 후 (실제): src 값이 https://image.ruliweb.com/.../<루리웹내부ID>.webp 형태
```

### 3. `img.alt`에 원본 업로드 파일명이 유지된다

`alt`에는 사용자가 첨부한 파일의 **원본 파일명**(확장자 포함)이 그대로 남습니다. 본 확장은 이 값을 매칭 키로 사용합니다.

```html
<img src="https://image.ruliweb.com/.../<id>.webp" alt="my_artwork_001.jpg">
```

### 4. 단일 / 다중 업로드 모두 `img.alt`가 유지된다

여러 파일을 동시에 업로드해도 각 `<img>`의 `alt`는 해당 원본 파일명을 유지합니다. 즉, 동일 페이지에 N개의 이미지가 있을 때 각 이미지의 `alt`는 사용자가 선택한 N개 파일의 이름과 일대일 대응합니다.

### 5. 한글, 공백, 괄호가 포함된 파일명도 그대로 유지된다

다음 같은 파일명도 인코딩 없이 `alt`에 보존됩니다.

- `루리웹 (1).png`
- `한글 파일명.jpg`
- `image (final) v2.webp`

따라서 매칭 시 `decodeURIComponent` 같은 추가 처리를 하지 않습니다. `File.name === img.alt`로 정확 비교합니다.

### 6. 업로드 실패 시 Progress 오브젝트만 남고 이미지 노드는 중복 생성되지 않는다

네트워크 실패 등으로 업로드가 실패하면 진행 중 표시(Progress)만 남고 `<img>`는 추가되지 않습니다. 동일 파일이 여러 `<img>`로 중복 생성되는 케이스는 관찰되지 않았습니다.

> 따라서 본 확장의 매칭 로직은 한 record당 한 번만 매칭을 시도하고 큐에서 제거하며, 재시도 로직은 두지 않습니다.

### 7. 직접 삽입한 출처 캡션 HTML은 source mode ↔ WYSIWYG mode 전환 후에도 유지된다

루리웹 에디터의 source mode와 WYSIWYG mode를 토글해도 다음 형태의 캡션은 보존됩니다.

```html
<p style="text-align: center;">
  출처: <a href="https://example.com/" target="_blank" rel="noopener noreferrer">https://example.com/</a>
</p>
```

이 사실은 본 확장이 캡션을 직접 삽입해도 사용자가 모드를 전환할 때 결과물이 살아남음을 의미합니다.

### 8. `href`, `target="_blank"`, `rel="noopener noreferrer"` 속성이 유지된다

위 7번에서 확인된 캡션의 속성들은 mode 전환 후에도 그대로 보존됩니다. 본 확장은 캡션 삽입 시 이 세 속성을 항상 함께 설정합니다.

## 매칭에 미치는 영향

| 관찰 사실 | 매칭 알고리즘에서의 적용 |
|---|---|
| 1, 4 | DOM 탐색은 `<p> > <img>` 패턴 한정 |
| 2 | `img.src` 비교 금지 |
| 3, 5 | `File.name === img.alt` 정확 비교 (인코딩 변환 없음) |
| 6 | 단일 매칭 1회 시도 후 큐에서 제거 (재시도 로직 없음) |
| 7, 8 | 캡션은 일반 HTML이며 보존 가능 — 사용자 작업물에 안전하게 남음 |

## 회귀 감지 방법 (제안 — Phase 2 이후)

루리웹 측 변경이 발생하면 위 8항 중 하나가 깨질 수 있습니다. Phase 2 이후 다음을 권고합니다.

- 첫 이미지 업로드 직후 DOM을 검사해 위 8항이 모두 성립하는지 가벼운 self-check를 수행한다.
- 깨진 항목이 있으면 캡션 삽입을 보류하고 옵션 페이지의 상태 영역(향후 추가)으로 사용자에게 알린다.
- self-check 결과는 콘솔에는 남기지 않는다 (사용자 페이지 콘솔 오염 방지). 옵션 페이지에서만 확인 가능하게 한다.

## 미관찰 / 결정 보류 항목

- 글쓰기 페이지 자체의 안정적인 selector (예: 본문 에디터 root) — Phase 2 진입 시 확인.
- `<img>`가 직접 본문 root 아래가 아닌 중첩된 `<div>` 안에 들어가는 경우의 처리 — Phase 2 진입 시 확인.
- 모바일 / 다른 게시판에서의 동일성 — 본 확장의 host_permissions를 확장하기 전에 별도 관찰 필요.
- iframe 형태의 에디터 사용 여부 — 사용 시 content script 주입 정책에 영향이 있으므로 Phase 2 진입 시 확인.
